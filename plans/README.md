## Vetted findings not (yet) planned

Ordered by leverage; ask to expand any into a plan.

1. **SSH connection reuse (PERF-01)** — every agent RPC spawns a fresh `ssh`
   process (full handshake per call; `internal/transport/ssh.go:37-82`,
   chokepoint `internal/agent/client.go:62`). Adding
   `ControlMaster/ControlPath/ControlPersist` to `args()` would cut deploy
   latency substantially. Effort S/M, risk LOW/MED (socket path limits,
   jump-host interaction). High-value next candidate.
2. **Parallelize per-host deploy execution (PERF-02)** — zero goroutines in
   the entire non-test codebase; `InspectObservedOnHosts` and
   `ExecuteActions` (`internal/deployment/deployment.go:154-169, 449-502`)
   iterate hosts serially, so wall-clock scales linearly with fleet size.
   Effort L, risk MED. Requires 001 (`-race` gate) first; pairs with the
   ControlMaster work above.
3. **Extract shared provider HTTP plumbing (DEBT-02/03)** — 12 of 26
   providers have a byte-identical `func (c Client) request(...)` (verified
   by signature grep: hetzner, digitalocean, vultr, linode, civo, cloudscale,
   exoscale, kamatera, oci, ovhcloud, scaleway, upcloud); none has
   retry/backoff, so any transient 429/5xx fails a whole provision. Extract
   once, then add bounded backoff in one place. Effort M/L, risk LOW.
4. **Split `internal/cli/root.go` (DEBT-01)** — 6,801 lines, ~13× the next
   largest file; every command and helper in one translation unit. Pure
   mechanical move into per-command files. Effort L, risk MED (import cycles,
   registration order).
5. **`internal/shipbinary` uses `curl` + unreachable fallback
   (DEPS-02/CORRECTNESS-05)** — `shipbinary.go:219-222` shells out to `curl`
   for the release download instead of `net/http` (the only such site in the
   repo), and the download fallback is unreachable when `crossCompile` fails
   for any reason other than "no module root" (`shipbinary.go:118-123`).
   Effort S. Note: the fallback also can't succeed until a release tag exists
   (see direction findings).
6. **Bump `golang.org/x/crypto`** — v0.45.0 → v0.53.0 available, on the
   age/secrets path. One-line bump; partially mitigated by 001's govulncheck
   gate. Effort S.
7. **Docs sync (DOCS-01)** — the env-level service/accessory deep-merge and
   the co-located-ingress alias upstreams are documented in the agent-facing
   `skills/ship/references/config-mapping.md` but absent from the human docs
   (`docs/configuration/README.md`, `docs/deploy-and-operate.md`). Also the

8. **Caddy startup grace tuning (CORRECTNESS-07, LOW confidence)** — the new
   `waitForCaddyContainer` gives Caddy 5s to reach Running before triggering

## Direction findings (maintainer decisions, not defects)

- **Cut a first release tag.** The repo has zero git tags; README already
  points users at `@v0.1.0 once releases are tagged`, the release workflow
  supports manual dispatch, and `shipbinary`'s download fallback hard-depends
  on a published release asset existing. Cheapest unlock on the list.
- **Reconcile PLAN.md.** Phases 6-11 contain many unchecked boxes
  (health gates, drain, rollback apply, deployment locks) for features the
  CLI now registers and Phase 13 declares complete — stale enough to mislead
  anyone (human or agent) using it as ground truth.
- **`ship import kamal`.** `skills/ship/references/migration-from-kamal.md`
  opens with "There is no `ship import kamal` command" and then specifies the
  full YAML mapping — the mapping table is effectively a spec awaiting a
  `ship import kamal <deploy.yml>` subcommand. Scope as a design/spike first.
- **Document ARM64 fleets.** The release matrix builds linux/arm64 and
  `shipbinary` cross-compiles per-platform, so heterogeneous/ARM hosts
  already work end-to-end — but no doc mentions it, and ARM instances are the
  cheapest offering on most of the 26 supported providers.

## Findings considered and rejected

- **Release tarball entry matching (`header.Name != "ship"`,
  `shipbinary.go:244`)**: rejected — the release workflow packs the binary as
  a bare `ship` entry (`tar -C dist -czf "$ARCHIVE" "${BINARY}"`,
  `.github/workflows/release.yml:157`), so the exact-name match is correct
  for Ship's own releases.
- **Agent RPC file-write/exec methods as a privilege issue**: rejected —
  the RPC rides the operator's SSH session (`ship agent rpc` over stdin);
  it grants nothing the SSH access doesn't already grant.
- **Proxmox `InsecureSkipVerify`**: rejected — gated behind an explicit
  `insecure_skip_tls_verify` opt-in; by-design.
- **Secrets in process args / `docker inspect`**: cleared — containers get
  secrets via `--env-file` with the remote file written `0o600`, not `-e`.
- **`accept-new` as the SSH default**: kept as default by design (fresh
  cloud hosts have unknowable keys); plan 005 only makes it overridable.
- **Shallow copy aliasing in `copyServices`/`copyAccessories`**: deferred,
  not rejected — latent hazard with no current mutation site; recorded in
  plan 003's maintenance notes.
